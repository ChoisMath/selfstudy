## 1. 데이터 모델 일관성

### 1-1. 역할 겸임 불가능한 단일 role 필드
- **심각도**: Critical
- **문제**: `teachers` 테이블의 `role` 필드가 단일 값(`admin/supervisor/homeroom`)으로 설계되어 있으나, 설계 문서 "역할 상세" 섹션에서 "한 교사가 여러 역할을 겸임할 수 있음 (예: 담임교사이면서 감독교사)"이라고 명시하고 있다. 단일 enum 필드로는 이를 지원할 수 없다.
- **근거**: 데이터 모델 `teachers.role (admin/supervisor/homeroom)` vs. 역할 상세 "한 교사가 여러 역할을 겸임할 수 있음"
- **제안**: 두 가지 방안 중 택일.
  - (A) `teacher_roles` 중간 테이블 도입: `teacher_id (FK)`, `role (enum)` -- 다대다 관계로 겸임 지원
  - (B) `role` 필드를 비트마스크 또는 배열(`role text[]`)로 변경 -- Prisma에서 PostgreSQL 배열 타입 지원 가능

### 1-2. 감독교사 날짜별/학년별 배정 테이블 부재
- **심각도**: Critical
- **문제**: 설계 문서에서 "감독교사는 관리자가 날짜별/학년별로 배정 (감독 일정 관리)"이라고 명시하지만, 이를 저장할 테이블이 데이터 모델에 존재하지 않는다. 상단 바에 "감독 김영수"를 표시하려면 해당 날짜에 어떤 교사가 어떤 학년의 감독인지 알 수 있어야 한다.
- **근거**: 역할 상세 "감독교사는 관리자가 날짜별/학년별로 배정" / UI 목업의 상단 바 "감독 김영수"
- **제안**: `supervisor_assignments` 테이블 추가:
  ```
  supervisor_assignments
  - id (PK)
  - teacher_id (FK)
  - date
  - grade
  - session_type (afternoon/night)
  - created_at
  ```

### 1-3. 학년 구분이 study_sessions에만 존재
- **심각도**: Warning
- **문제**: `study_sessions` 테이블에 `grade` 필드가 있지만, `attendance` 테이블에는 학년 정보가 없다. 학년별 출석 통계를 조회하려면 `attendance -> student -> grade`로 JOIN해야 하며, 이는 대량 데이터 조회 시 성능 저하를 유발할 수 있다. 또한 `seat_layouts`에도 학년 정보가 직접 없어 `room -> session -> grade` 경로로 접근해야 한다.
- **근거**: 관리자 기능 "학년 선택 -> 해당 학년의 자습 관리", "기간별 출석 통계"
- **제안**: `attendance` 테이블에 `grade` 필드를 비정규화하여 추가하거나, 최소한 복합 인덱스 `(date, session_type)`와 함께 student JOIN 최적화를 계획에 포함할 것.

### 1-4. attendance 테이블 인덱스 설계 부재
- **심각도**: Warning
- **문제**: `attendance` 테이블은 가장 빈번하게 조회되는 테이블(매일 출석체크, 주간 조회, 기간별 통계)인데, 인덱스 설계가 전혀 언급되지 않았다. 주간 출석 팝업 조회 시 `WHERE student_id = ? AND date BETWEEN ? AND ?` 쿼리가 매 더블클릭마다 발생한다.
- **근거**: 감독교사 화면의 "더블클릭 -> 주간 출석현황 팝업", 관리자 "기간별 출석 통계"
- **제안**: 최소한 다음 인덱스를 설계 문서에 명시:
  - `(student_id, date)` -- 주간/기간별 조회
  - `(date, session_type)` -- 일별 전체 출석 조회
  - `(student_id, session_type, date)` UNIQUE -- 중복 출석 방지

### 1-5. attendance 레코드 유일성 제약 부재
- **심각도**: Critical
- **문제**: 같은 학생이 같은 날짜, 같은 세션에 대해 중복 attendance 레코드가 생성될 수 있다. UNIQUE 제약이 명시되지 않았다.
- **근거**: 데이터 모델 `attendance` 테이블 정의
- **제안**: `(student_id, session_type, date)` 복합 UNIQUE 제약 추가.

## 2. 기능 누락

### 2-1. 비밀번호 변경/초기화 기능 부재
- **심각도**: Warning
- **문제**: 인증 섹션에서 "기본: ID/비밀번호 (관리자가 계정 생성)"이라고 되어 있으나, 교사가 비밀번호를 변경하거나 관리자가 초기화하는 기능이 화면 설계에 없다.
- **근거**: 인증 섹션, 화면 설계의 관리자 화면/감독교사 화면/담임교사 화면 어디에도 비밀번호 관련 UI 없음
- **제안**: 최소한 관리자가 비밀번호를 초기화하는 기능과, 교사 본인이 비밀번호를 변경하는 설정 화면을 추가.

### 2-2. 학기/기간 관리 기능 부재
- **심각도**: Warning
- **문제**: UI 목업의 주간 팝업에 "4월 1주차"라는 표기가 있고, 좌석 배치는 기간에 따라 바뀔 수 있다(예: 3-4월 좌석 배치). 그러나 `seat_layouts` 테이블에 기간 정보가 없고, 학기/기간을 관리하는 테이블도 없다. 좌석 배치를 변경하면 이전 배치가 덮어씌워져 과거 출석 기록과 좌석 위치의 연관이 깨진다.
- **근거**: 관리자 "좌석 배치 편집", UI 목업 "4월 1주차"
- **제안**: `seat_layouts`에 `period_start`, `period_end` 필드를 추가하거나, `seating_periods` 테이블을 별도로 만들어 좌석 배치의 이력 관리를 지원. 최소한 좌석 배치 변경 시 스냅샷을 보관하는 방안 필요.

### 2-3. 학년 전환(연도 변경) 시 데이터 마이그레이션 미고려
- **심각도**: Suggestion
- **문제**: 새 학년이 시작되면 학생의 학년/반이 바뀌고, 졸업생은 비활성화되어야 한다. 이에 대한 계획이 없다.
- **근거**: `students` 테이블에 학년도(year) 필드 없음, 활성/비활성 상태 없음
- **제안**: MVP에서는 수동으로 처리하되, `students` 테이블에 `year` 또는 `is_active` 필드 추가를 고려. 최소한 "학년 전환 시 어떻게 할 것인가"에 대한 운영 절차를 문서화.

### 2-4. 감독교사 배정 관리 화면 부재
- **심각도**: Warning
- **문제**: "감독교사는 관리자가 날짜별/학년별로 배정"한다고 했으나, 관리자 화면 설계에 감독교사 배정 관리 UI가 명시되지 않았다. 교사 명단 CRUD와 담임교사 배정만 있다.
- **근거**: 관리자 화면 목록에 감독교사 일정/배정 관리 항목 없음
- **제안**: 관리자 화면에 "감독교사 일정 관리" 메뉴 추가. 달력 또는 주간 테이블 형태로 날짜별/학년별 감독교사를 배정하는 UI 필요.

## 3. UI/UX 문제

### 3-1. 모바일에서 클릭과 더블클릭/길게 누르기 충돌
- **심각도**: Critical
- **문제**: 모바일 터치 환경에서 "탭 = 출석 토글"과 "더블탭/길게 누르기 = 주간 팝업"의 구분이 까다롭다. 더블탭 시 첫 번째 탭이 출석 토글로 먼저 처리되고, 두 번째 탭으로 다시 상태가 변경된 후 팝업이 뜨는 문제가 발생한다. 길게 누르기(long press)의 경우 OS 기본 동작(텍스트 선택, 컨텍스트 메뉴)과 충돌할 수 있다.
- **근거**: 좌석 인터랙션 "클릭(탭): 출석 상태 순환 토글", "더블클릭(길게 누르기): 주간 출석현황 팝업"
- **제안**: 세 가지 대안 중 택일:
  - (A) 탭 = 출석 토글, 길게 누르기(300ms+) = 팝업. 탭 처리를 300ms 지연시켜 길게 누르기와 구분 (단, 반응성 저하)
  - (B) 탭 = 출석 토글, 별도 정보 아이콘(i) 탭 = 팝업 (가장 안전한 방법)
  - (C) 스와이프로 팝업 열기

### 3-2. 50석 미래예습실1 도면 레이아웃의 기술적 난이도
- **심각도**: Warning
- **문제**: 야간자습 도면 기반 레이아웃에서 미래예습실1은 50석 규모이다. 모바일 화면(375px)에서 50개 좌석을 격자로 표현하면 각 좌석이 매우 작아지거나(이름 판독 불가), 가로 스크롤이 필요하다. UI 목업은 오후자습(15석 교실)만 구현되어 있고, 야간자습 50석 레이아웃은 검증되지 않았다.
- **근거**: 야간자습 "미래예습실1(50석)", UI 목업에 야간자습 탭 미구현
- **제안**: 50석 레이아웃에 대한 별도 목업 필요. 가능한 접근법:
  - 10열x5행 격자에서 이름 축약(성+이름 첫 글자) + 확대/축소 제스처
  - 영역 분할 (좌측 25석 / 우측 25석으로 두 그리드)
  - 가로 모드 권장 안내

### 3-3. 야간자습 도면 레이아웃의 구체적 구현 방법 미명시
- **심각도**: Warning
- **문제**: "도면 기반 레이아웃: 실제 미래홀 구조 반영"이라고만 되어 있고, 5개 방(미래예습실1, 미래예습실2, 미래202, 미래아띠존, 미래201)을 어떻게 배치할지 구체적 와이어프레임이 없다. 교실별 카드 나열인지, 실제 건물 평면도처럼 2D 배치인지 불분명하다.
- **근거**: 야간자습 "도면 기반 레이아웃" 설명
- **제안**: MVP에서는 오후자습과 동일하게 방별 카드 나열(세로 스크롤)로 구현하고, 도면 기반 2D 배치는 후순위로 미룰 것을 권장. 도면 기반 배치가 필수라면 구체적인 와이어프레임과 좌표 시스템 설계가 필요.

### 3-4. 좌석 배치 드래그&드롭의 모바일 동작
- **심각도**: Suggestion
- **문제**: 관리자 좌석 배치 편집에서 "드래그&드롭"이라고 했지만, 모바일 브라우저에서 HTML5 드래그&드롭 API는 기본적으로 동작하지 않는다.
- **근거**: 관리자 화면 "학생을 좌석에 드래그&드롭 배치"
- **제안**: 관리자는 주로 데스크톱에서 작업할 가능성이 높으므로 심각하지는 않지만, 모바일 대응이 필요하다면 `react-dnd` + `react-dnd-touch-backend` 또는 `@dnd-kit/core`(터치 지원 내장) 사용을 명시. 대안으로 모바일에서는 "학생 선택 -> 좌석 선택" 두 단계 방식도 가능.

## 4. 보안 및 인증

### 4-1. Credentials Provider 보안 고려사항 미흡
- **심각도**: Warning
- **문제**: NextAuth.js의 Credentials Provider는 세션 관리를 JWT로 처리하며, 기본적으로 CSRF 보호가 제한적이다. 비밀번호 해싱 알고리즘(bcrypt 등), 로그인 시도 제한(brute force 방지), JWT 만료 시간 등의 보안 설정이 설계에 없다.
- **근거**: 인증 섹션 "기본: ID/비밀번호", 기술 스택 "NextAuth.js (Credentials + Google Provider)"
- **제안**: 설계 문서에 최소한 다음을 명시:
  - 비밀번호 해싱: bcrypt (cost factor 12+)
  - JWT 만료: 적절한 세션 수명 (예: 8시간, 학교 일과 시간 기준)
  - 로그인 실패 제한: 5회 실패 시 일시 잠금 (또는 rate limiting)

### 4-2. RBAC 구현 방법 미명시
- **심각도**: Warning
- **문제**: 역할별 권한이 표로 정의되어 있지만, 실제 API 라우트에서 어떻게 역할을 확인하고 접근을 제어할지 구현 방법이 없다. 감독교사가 관리자 API를 호출하거나, 담임교사가 다른 반 학생 데이터에 접근하는 것을 어떻게 막을지 불분명하다.
- **근거**: 사용자 역할 및 권한 표, 화면 설계 전반
- **제안**: Next.js 미들웨어 또는 API Route Handler에서의 인가 패턴을 명시:
  - `middleware.ts`에서 역할 기반 라우트 보호 (예: `/admin/*`은 admin만)
  - API Route에서 세션 확인 + 역할 확인 유틸리티 함수
  - 담임교사의 데이터 접근 범위 제한 (자기 반 학생만)

### 4-3. API 라우트 인증/인가 미들웨어 미계획
- **심각도**: Warning
- **문제**: API Routes(Route Handlers)에 대한 인증/인가 미들웨어 계획이 없다. 모든 API 엔드포인트 목록도 없고, 각 엔드포인트에 필요한 역할도 정의되지 않았다.
- **근거**: 기술 스택 "Next.js API Routes (Route Handlers)", 전체 설계 문서에 API 엔드포인트 정의 부재
- **제안**: 주요 API 엔드포인트 목록과 각각의 필요 역할을 정의. 예:
  - `GET /api/attendance?date=&session=` -- supervisor, admin
  - `PUT /api/attendance/:id` -- supervisor, admin
  - `POST /api/absence-reasons` -- homeroom, admin
  - `GET /api/students?class=` -- homeroom(자기 반), admin(전체)

## 5. 기술적 위험

### 5-1. 실시간 출석 상태 동기화 미고려
- **심각도**: Warning
- **문제**: 여러 감독교사가 동시에 같은 학년의 출석을 체크할 수 있다(예: 한 교사가 2-4반, 다른 교사가 2-5반을 동시에). Next.js App Router에서 실시간 동기화 방법(폴링, SSE, WebSocket)이 명시되지 않았다. 한 교사가 체크한 결과가 다른 교사의 화면에 반영되지 않을 수 있다.
- **근거**: 감독교사 출석체크 화면, 상단 바의 "출석 N 결석 N 미체크 N" 실시간 카운트
- **제안**: MVP에서는 주기적 폴링(30초 간격)으로 충분. SWR의 `refreshInterval` 또는 React Query의 `refetchInterval` 활용. 향후 필요 시 Server-Sent Events 고려.

### 5-2. 동시성 제어(Optimistic Concurrency) 부재
- **심각도**: Warning
- **문제**: 같은 학생의 출석을 두 교사가 동시에 토글하면 나중 요청이 이전 요청을 덮어쓴다(lost update). `attendance` 테이블에 `updated_at` 또는 버전 필드가 없어 충돌 감지가 불가능하다.
- **근거**: 데이터 모델 `attendance` 테이블에 updated_at 없음
- **제안**: 실제 운영에서는 한 학년에 한 명의 감독교사가 배정되므로 동시성 문제가 드물 수 있다. 그러나 최소한 `attendance` 테이블에 `updated_at` 필드를 추가하고, Prisma의 `@updatedAt` 어노테이션 사용을 권장.

### 5-3. Railway Volume 용도 불명확
- **심각도**: Suggestion
- **문제**: 기술 스택에 "Storage: Railway Volume"이 있지만, 이 시스템에서 파일 스토리지가 필요한 기능이 명확하지 않다. Excel 다운로드는 서버에서 생성 후 바로 스트리밍하면 되므로 별도 Volume이 필요하지 않다.
- **근거**: 기술 스택 "Storage: Railway Volume"
- **제안**: Volume의 구체적 용도를 명시하거나, 불필요하면 제거. 만약 Excel 파일 캐싱이나 로그 저장 용도라면 그 목적을 문서화.

### 5-4. SheetJS 커뮤니티 버전 라이센스 제한
- **심각도**: Warning
- **문제**: 설계에서 "xlsx 라이브러리 (SheetJS)"를 언급하나, SheetJS Community Edition은 Apache 2.0 라이센스이지만 npm의 `xlsx` 패키지는 2023년 이후 일부 기능이 Pro 버전으로 이동했다. `.xlsx` 쓰기 기능이 커뮤니티 버전에서 제한될 수 있다.
- **근거**: 기술 스택 "xlsx 라이브러리 (SheetJS)"
- **제안**: 대안 라이브러리 검토:
  - `exceljs` -- MIT 라이센스, .xlsx 읽기/쓰기 모두 무료, 스타일링 지원
  - `xlsx-populate` -- MIT 라이센스
  - 실제로 `xlsx` 커뮤니티 버전으로 필요한 기능이 가능한지 사전 PoC 필요

## 6. 과잉 설계 또는 미흡한 설계

### 6-1. 도면 기반 레이아웃은 과잉 설계
- **심각도**: Suggestion
- **문제**: 야간자습의 "도면 기반 레이아웃"은 기술적 난이도가 높고, MVP에서 필수적이지 않다. 오후자습과 동일한 카드 나열 방식으로도 충분히 기능한다.
- **근거**: 야간자습 "도면 기반 레이아웃: 실제 미래홀 구조 반영"
- **제안**: Phase 1에서는 방별 카드 나열, Phase 2에서 도면 기반 레이아웃으로 진화.

### 6-2. study_sessions 테이블의 과잉 설계
- **심각도**: Suggestion
- **문제**: `study_sessions` 테이블에 `time_start`, `time_end`가 있지만, 시간은 고정값(오후 16:30~18:20, 야간 19:20~21:00)이며 변경될 일이 거의 없다. 세션 타입이 사실상 afternoon/night 두 가지뿐이라면 별도 테이블 대신 enum으로 충분할 수 있다.
- **근거**: `study_sessions` 테이블 정의, 오후/야간 고정 시간
- **제안**: 유지해도 무방하지만, 초기 시드 데이터로 고정 세션을 넣어두고 런타임에서 변경하지 않는 방식이면 테이블 없이 상수로 처리해도 된다. rooms 테이블이 session_id에 의존하므로 현재 구조를 유지하되, `rooms.session_type (afternoon/night)` + `rooms.grade`로 단순화하는 것도 고려.

### 6-3. participation_days의 미흡한 설계
- **심각도**: Warning
- **문제**: `participation_days` 테이블에 기간 정보가 없다. 학생의 참석요일이 학기 중간에 바뀔 수 있는데(예: 4월부터 학원 등록으로 수요일 불참), 변경 이력을 추적할 수 없다. 과거 데이터를 분석할 때 "이 학생이 그 당시 참석 대상이었는지" 판단이 불가능하다.
- **근거**: 담임교사 "학생별 참석요일 설정 (월~금 토글, 오후/야간 각각)", `participation_days` 테이블
- **제안**: MVP에서는 현재 설계로 시작하되, `effective_from` 날짜 필드 추가를 고려. 변경 시 새 레코드를 생성하는 방식으로 이력 관리.

## 7. 구현 실행 순서

### 7-1. 구현 순서의 의존성 오류
- **심각도**: Critical
- **문제**: 설계 문서에는 명시적 구현 순서가 없지만, 검증 계획의 흐름을 보면 감독교사 -> 담임교사 -> 관리자 순으로 개발하는 것이 암시된다. 그러나 실제 의존성은:
  1. 관리자가 먼저 학생 명단을 등록해야 함
  2. 관리자가 좌석을 배치해야 함
  3. 관리자가 교사 계정을 생성하고 감독교사를 배정해야 함
  4. 그래야 감독교사 화면이 데이터와 함께 동작함
  
  즉, **관리자 기능이 먼저** 구현되어야 한다.
- **근거**: 검증 계획 순서, 화면 설계 간 데이터 의존성
- **제안**: 올바른 구현 순서:
  1. **Phase 0**: 데이터 모델 + 시드 데이터 + 인증
  2. **Phase 1**: 관리자 -- 학생/교사 CRUD, 좌석 배치 편집 (최소한 수동 입력)
  3. **Phase 2**: 감독교사 -- 출석체크 화면 (Phase 1에서 입력한 데이터 기반)
  4. **Phase 3**: 담임교사 -- 학생 관리, 불참사유 등록
  5. **Phase 4**: 관리자 통계 + Excel 다운로드
  
  대안: 시드 스크립트로 테스트 데이터를 미리 넣어두면 감독교사 화면을 먼저 개발할 수도 있지만, 이 경우 시드 스크립트 작성이 Phase 0에 포함되어야 한다.

## 종합 평가

| 항목 | 평가 |
|------|------|
| **전반적 품질** | **중** |
| **Critical 이슈** | **4건** (역할 겸임 불가, 감독교사 배정 테이블 부재, 출석 유일성 제약 부재, 모바일 터치 충돌, 구현 순서 오류) |
| **Warning 이슈** | **11건** |
| **Suggestion 이슈** | **4건** |

### 판단

설계의 전반적 방향은 적절하다. 학교 출석 관리라는 도메인을 잘 이해하고 있으며, 역할 분리, 화면 설계, 기술 스택 선택이 합리적이다. UI 목업의 반응형 설계(clamp, sticky header, 그리드)도 실용적이다.

그러나 **데이터 모델에 4건의 Critical 이슈**가 있어 현재 상태로는 구현에 착수할 수 없다. 특히:

1. **역할 겸임을 지원하는 데이터 구조**가 없으면 핵심 요구사항을 충족할 수 없다.
2. **감독교사 배정 테이블**이 없으면 감독교사 화면의 핵심 기능("오늘 내가 감독할 학년" 표시)이 불가능하다.
3. **모바일 터치 인터랙션** 설계가 해결되지 않으면 주 사용 시나리오(모바일로 출석체크)에서 UX 문제가 발생한다.

**권장 사항**: Critical 이슈 4건을 먼저 해결하고, 구현 순서를 재조정한 후 구현에 착수할 것. 데이터 모델 수정은 코드 작성 전에 완료하는 것이 비용이 가장 낮다.
