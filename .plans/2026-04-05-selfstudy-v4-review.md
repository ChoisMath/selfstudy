## 1. 데이터 모델 일관성

### 1-1. sub_admin 역할과 SubAdminAssignment의 중복 구조
- **심각도**: Warning
- **문제**: `TeacherRole`에 `sub_admin` 역할을 추가하고, 동시에 `SubAdminAssignment` 테이블로 학년을 매핑하는 이중 구조. 서브관리자인지 확인하려면 `teacher_roles`에서 `sub_admin` 역할 존재 여부를 확인한 뒤, 다시 `sub_admin_assignments`에서 학년을 조회해야 한다. 두 테이블 간 정합성이 깨질 수 있다 -- 예: `teacher_roles`에 `sub_admin`이 있지만 `sub_admin_assignments`에 행이 없는 경우, 또는 그 반대.
- **근거**: 계획서 3절 데이터 모델 -- `TeacherRole` enum에 `sub_admin` 존재 + `SubAdminAssignment` 모델 별도 존재
- **제안**: 두 가지 접근 중 하나를 선택하라:
  - **(A) SubAdminAssignment만 사용**: `teacher_roles`에서 `sub_admin` 역할을 제거하고, `SubAdminAssignment` 존재 여부로 서브관리자를 판별. 역할 체크 미들웨어에서 `sub_admin_assignments` 테이블을 JOIN.
  - **(B) teacher_roles만 사용**: `SubAdminAssignment`를 제거하고, `TeacherRole`에 `grade` 필드를 추가하여 `sub_admin` 역할에 학년 정보를 포함.
  - 어느 쪽이든 단일 소스 오브 트루스(Single Source of Truth)를 유지해야 한다.
  - 만약 현재 구조를 유지한다면, 서브관리자 지정/해제 시 양쪽 테이블을 트랜잭션으로 동기화하는 로직이 반드시 필요하며, 이를 계획서에 명시해야 한다.

### 1-2. 학생 로그인 시 동명이인 구분 로직의 모호함
- **심각도**: Warning
- **문제**: 계획서 10절에서 "이름+학번으로 유일하게 식별 가능"이라고 했으나, 로그인 폼에서 ID=이름, PW=학번이므로 문제가 없어 보인다. 그러나 5절 보안 설계에서는 "동명이인: 이름 입력 후 학년/반 선택 화면"이라는 다른 흐름을 제시하고 있다. 두 가지 설명이 상충한다.
  - 시나리오: 1학년 김서현(10102), 2학년 김서현(20305). 학생이 이름 "김서현" + 학번 "20305"를 입력하면 → DB에서 `name="김서현" AND 학번=20305`로 단일 매칭 가능. 이 경우 학년/반 선택 화면은 불필요.
  - 반대로 학년/반 선택 화면이 있다면, 학번 입력이 왜 필요한가?
- **근거**: 계획서 5절 "동명이인: 이름 입력 후 학년/반 선택 화면 → 특정 학생 확인" vs 10절 "이름+학번으로 유일하게 식별 가능"
- **제안**: 하나의 흐름으로 통일하라.
  - **권장안**: 이름(ID) + 학번(PW) 입력만으로 로그인 완료 (학번이 사실상 고유 식별자이므로 학년/반 선택 화면은 불필요). 5절의 동명이인 처리 설명을 삭제하거나, "학번으로 유일 식별되므로 추가 선택 불필요"로 수정.
  - 또는 학번을 ID로, 이름을 PW로 뒤집는 것도 고려 (학번이 고유하므로 더 자연스러운 ID).

### 1-3. AbsenceRequest 승인 시 Attendance/AbsenceReason 연동 로직 미명시
- **심각도**: Critical
- **문제**: 학생이 불참 신청 → 담임 승인 시, 해당 날짜의 `Attendance` 레코드를 `absent`로 설정하고 `AbsenceReason`을 생성해야 한다. 그러나 계획서에는 이 연동 로직이 어디에도 명시되어 있지 않다. API 엔드포인트 `PUT /api/homeroom/absence-requests/:id` (승인/반려)가 내부적으로 어떤 작업을 수행하는지 불분명하다.
  - 추가 문제: 승인된 불참이 나중에 번복되면? Attendance 레코드도 롤백해야 하는가?
  - `AbsenceReason.registeredBy`는 교사 ID인데, 학생 신청 → 담임 승인 경로에서는 누가 registrar가 되는가?
- **근거**: 계획서 2-2절 "승인 시 출석에 반영" (한 줄 언급), API 테이블의 `PUT /api/homeroom/absence-requests/:id`
- **제안**:
  - 승인 API의 트랜잭션 로직을 명확히 기술하라: (1) `AbsenceRequest.status` → `approved` (2) `Attendance` upsert (`absent`) (3) `AbsenceReason` 생성 (`registeredBy` = 승인 교사, `reasonType`/`detail` = 신청 내용 복사)
  - 반려 시에는 Attendance 변경 없음을 명시
  - 승인 후 번복 가능 여부와 그 처리도 정의 필요

### 1-4. Attendance 레코드 생성 시점 미정의
- **심각도**: Warning
- **문제**: `Attendance` 레코드가 언제 생성되는가? 감독교사가 좌석을 탭할 때 최초 생성(upsert)인가, 아니면 매일 배치로 미리 생성하는가? 불참 승인 시에도 Attendance를 생성해야 하는데, 이미 존재할 수도 있고 없을 수도 있다.
- **근거**: 계획서 전반에 Attendance 레코드 생성 시점에 대한 언급 없음
- **제안**: "출석체크 시 upsert" 방식을 명시하고, 불참 승인 시에도 동일한 upsert 로직을 사용하도록 설계. 또는 당일 자습 시작 시 참여 예정 학생에 대해 `unchecked` 상태로 일괄 생성하는 배치를 고려.

## 2. 권한/보안

### 2-1. 서브관리자 학년 파라미터 조작 취약점
- **심각도**: Critical
- **문제**: `/grade-admin/[grade]/*` 라우트에서 `grade`는 URL 파라미터이다. 2학년 서브관리자가 URL을 `/grade-admin/1/students`로 수동 변경하면 1학년 데이터에 접근할 수 있다. 계획서의 RBAC 테이블에서는 `sub_admin (해당 학년만)`이라고 했지만, 실제 미들웨어 구현에서 `SubAdminAssignment.grade`와 URL의 `grade` 파라미터를 대조하는 로직이 누락될 위험이 높다.
- **근거**: 계획서 5절 인가 테이블 "sub_admin (해당 학년만)", 4절 서브관리자 라우트 구조
- **제안**:
  - 미들웨어에서 반드시 세션의 `teacherId` → `SubAdminAssignment` 조회 → `grade` 일치 여부를 검증하는 로직을 명시하라.
  - API 레벨에서도 동일한 검증 필요 (프론트엔드 라우팅 보호만으로는 불충분).
  - 구현 계획에 "학년 파라미터 검증 미들웨어"를 Phase 0의 RBAC 미들웨어 항목에 포함시키라.

### 2-2. 학생 로그인 보안 취약성
- **심각도**: Warning
- **문제**: 학번은 공개 정보에 가깝다 (학년+반+번호는 교실에 게시되는 정보). 이름도 공개 정보이다. 따라서 학생 로그인은 사실상 "공개정보+공개정보"로 인증하는 셈이며, 같은 학교 학생이라면 누구의 계정이든 로그인 가능하다.
- **근거**: 계획서 1절 학생 로그인 규칙, 5절 보안 설계 "학번은 해싱하지 않고 DB 검증"
- **제안**:
  - 학생이 할 수 있는 행위가 "불참 신청"뿐이고, 승인은 담임이 하므로 공격 영향도는 낮다. 이 위험을 수용한다면 계획서에 "학생 인증은 편의성 우선, 보안 수준 낮음 (허용됨)"을 명시적으로 기재하라.
  - 추가 보호 방안으로 학생 세션 만료를 짧게 (예: 1시간) 설정하고, IP/디바이스 기반 이상 접근 감지는 과잉이므로 불필요.
  - Rate limiting (5회 실패 시 잠금)은 교사에만 언급되어 있는데, 학생 로그인에도 brute force 방지를 적용해야 한다 (전수 대입 가능: 이름 알면 학번 5자리 10만 조합이지만, 학번 형식이 `[1-3][01-20][01-40]`이므로 실제 유효 조합은 2400개 미만).

### 2-3. 감독교체 시 상대 교사 동의 없는 일방적 교체
- **심각도**: Warning
- **문제**: 계획서의 교체 프로세스(2-4절)에서 교사 A가 교사 B를 선택하고 "확인"을 누르면 즉시 교체가 완료된다. 교사 B의 동의 절차가 없다. 교사 B는 본인도 모르게 감독 일정이 추가될 수 있다.
- **근거**: 계획서 2-4절 "교사가 '확인' 클릭 시 교체 진행 → supervisor_assignments 업데이트", 8-3절 교체 UI
- **제안**:
  - 현재 학교 업무 관행상 교체가 사전 합의 후 시스템에 기록하는 것이라면, 현 구조로 충분하다 (오프라인 합의 전제). 이 경우 계획서에 "교체는 당사자 간 사전 합의를 전제로 하며, 시스템은 기록 목적"임을 명시하라.
  - 만약 시스템 내 동의가 필요하다면, `supervisor_swap_request` 테이블 추가 + 상대 교사 승인 워크플로우가 필요하나, 이는 복잡도가 크게 증가하므로 MVP에서는 비권장.

## 3. 기능 누락/모호함

### 3-1. Excel 업로드 시 학생 학번(PW) 자동 생성 여부
- **심각도**: Warning
- **문제**: 서브관리자/메인관리자가 학생을 Excel로 일괄 업로드할 때, 학번(=비밀번호)은 `grade*10000 + classNumber*100 + studentNumber`로 자동 계산되어야 한다. 그러나 이것이 업로드 시 자동 생성인지, Excel에 학번 열이 있어야 하는지 불분명하다.
- **근거**: 계획서 2-6절 "학생 명단: Excel 파일(학년, 반, 번호, 이름) 업로드", 10절 학번 계산 규칙
- **제안**: Excel 템플릿에는 학번 열 없이 (학년, 반, 번호, 이름)만 포함하고, 서버 측에서 학번을 자동 계산하여 로그인에 사용함을 명시하라. Student 모델에 `studentCode` 같은 필드가 없으므로, 로그인 시 매번 계산하거나 가상 필드로 처리해야 한다 -- 이 로직도 명시 필요.

### 3-2. 한 교사가 여러 학년의 서브관리자가 될 수 있는지
- **심각도**: Suggestion
- **문제**: `SubAdminAssignment`의 유니크 제약이 `@@unique([teacherId, grade])`이므로, 한 교사가 1학년+2학년 서브관리자를 겸임할 수 있다. 의도된 설계인지 불분명.
- **근거**: 계획서 3절 SubAdminAssignment 모델
- **제안**: 의도된 것이라면 네비게이션에 "1학년 데이터관리", "2학년 데이터관리" 탭이 복수로 표시되어야 한다. 의도가 아니라면 `@@unique([teacherId])` 또는 `@@unique([grade])`로 제약을 변경하라. 어느 쪽이든 계획서에 명시하라.

### 3-3. 감독교체 시 교체 대상 교사 선택 범위
- **심각도**: Warning
- **문제**: 교체 대상 교사 목록의 범위가 불분명하다. "같은 학년 + 다른 학년 모두 가능"이라 했지만, 선택 가능한 교사는 모든 교사인가? `supervisor` 역할이 있는 교사만인가? 해당 날짜에 이미 감독 배정이 있는 교사는 제외되는가?
- **근거**: 계획서 2-4절 "교체 대상 교사 B를 선택 (같은 학년 + 다른 학년 모두 가능)"
- **제안**: 선택 가능 조건을 명확히 정의하라:
  - 예: "supervisor 또는 homeroom 역할을 가진 모든 교사" 또는 "해당 날짜에 감독 배정이 없는 교사"
  - 이미 감독인 교사와 교체하면 "교환(swap)"인지 "일방 이전"인지도 정의 필요

### 3-4. 월간 출결 달력에서 오후/야간 구분 표시
- **심각도**: Suggestion
- **문제**: 학생 월간 달력 뷰(8-4절)에서 하루에 오후/야간 두 세션이 있으므로, 한 칸에 두 상태를 어떻게 표시할지 정의되지 않았다.
- **근거**: 계획서 8-4절 "이번 달 → 달력 형태 표시, 출석일=초록, 결석일=빨강"
- **제안**: 달력 셀을 상/하 또는 좌/우로 분할하여 오후(상단)/야간(하단) 표시. 또는 세션 탭(오후/야간)으로 분리. 이 UI 설계를 명시하라.

### 3-5. 메인관리자와 서브관리자 코드 재사용 전략 부재
- **심각도**: Warning
- **문제**: `/admin/*`와 `/grade-admin/[grade]/*`는 학생관리, 참가설정, 좌석배치, 감독배정 등 상당 부분이 겹친다. 계획서에 컴포넌트 재사용 전략이 없으며, Phase 1과 Phase 1.5에서 동일 기능을 두 번 구현할 위험이 있다.
- **근거**: 계획서 7절 Phase 1(메인관리자) vs Phase 1.5(서브관리자) -- 학생 CRUD, Excel 업로드, 참가설정, 좌석배치, 감독배정이 양쪽에 중복
- **제안**:
  - 공통 컴포넌트를 `components/admin-shared/` 디렉토리에 추출하고, `grade` prop으로 필터링 범위만 다르게 전달하는 구조를 명시하라.
  - API도 `/api/grade-admin/[grade]/*` 하나로 통합하고, 메인관리자는 모든 grade에 접근 가능하게 하는 것이 깔끔하다.
  - Phase 1.5를 별도로 두지 말고, Phase 1에서 학년별 컴포넌트로 설계한 뒤 메인관리자/서브관리자의 차이는 권한 레이어에서 처리하라.

## 4. UX/UI 문제

### 4-1. 역할 겸임 시 네비게이션 과부하
- **심각도**: Warning
- **문제**: 한 교사가 담임+감독+서브관리자를 겸임하면 네비게이션 탭이 최소 3~4개 이상 (출석체크, 내 반 관리, OO학년 데이터관리, 감독일정 등). 모바일에서 탭 바가 넘칠 수 있다.
- **근거**: 계획서 1절 역할 겸임 "한 교사가 여러 역할 겸임 가능"
- **제안**:
  - 햄버거 메뉴 또는 "더보기" 드롭다운으로 보조 기능을 숨기는 방안을 설계하라.
  - 역할별 우선순위를 정하여 주 탭과 보조 메뉴를 분리 (예: 오늘 감독이면 출석체크가 메인, 나머지는 메뉴).

### 4-2. 학생 로그인 시 이름 입력 UX
- **심각도**: Suggestion
- **문제**: 학생이 이름을 자유 입력하는데, 오타 시 로그인 실패한다. 자동완성을 제공하면 학생 명단이 노출되는 보안 문제가 있다.
- **근거**: 계획서 8-1절 학생 로그인 탭
- **제안**: 자동완성은 제공하지 말고, 로그인 실패 시 "이름 또는 학번이 일치하지 않습니다" 일반 에러 메시지 사용. 이름 대신 학번만으로 로그인하는 대안도 고려 (학번이 사실상 고유 ID이므로 ID+PW 모두 학번으로 통일하면 이름 입력 오류 자체가 사라짐 -- 단, UX 관점에서 "이름"이 더 친숙).

### 4-3. 타학년 교체 확인 모달의 동의 절차 부재
- **심각도**: Suggestion
- **문제**: 4-3은 2-3과 동일한 문제의 UX 측면. 모달에서 "진행하시겠습니까?"는 요청자(교사 A)에게만 확인을 구하며, 피교체자(교사 B)는 알림조차 받지 못한다.
- **근거**: 계획서 2-4절 확인 모달 UI
- **제안**: 최소한 교체 후 교사 B가 다음 로그인 시 "감독 일정이 변경되었습니다" 알림을 볼 수 있게 하라. 이는 별도 알림 시스템 없이, 감독교사 화면의 상단 바에서 일정 변경을 표시하는 것으로 간단히 구현 가능.

## 5. 구현 복잡도/과잉설계

### 5-1. 역할 + 학년별 권한의 미들웨어 복잡도
- **심각도**: Warning
- **문제**: 5종 역할(admin, sub_admin, supervisor, homeroom, student) + 역할 겸임 + 서브관리자의 학년별 접근 제어 = 미들웨어가 매우 복잡해진다. 특히 `sub_admin`은 역할 확인 + 학년 파라미터 대조가 필요하고, `homeroom`은 역할 확인 + 담당반 대조가 필요하다.
- **근거**: 계획서 5절 인가 테이블, 1절 역할 정의
- **제안**:
  - 권한 체크를 2단계로 분리: (1) 역할 기반 접근 (라우트 레벨) → (2) 데이터 범위 필터링 (API 레벨). 이를 `withRoleAuth(roles)` + `withScopeFilter(scopeType)` 두 개의 래퍼로 구현.
  - 가능하면 `student` 역할은 NextAuth의 별도 Provider로 완전 분리하여 교사 권한 체계와 섞이지 않게 하라.

### 5-2. Phase 축소 가능성 (MVP 범위)
- **심각도**: Suggestion
- **문제**: Phase 0~5까지 6단계이며, 각 Phase의 작업량이 상당하다. 특히 Phase 1.5(서브관리자)와 Phase 4(학생)는 v4에서 새로 추가된 것으로, 전체 구현 기간이 길어진다.
- **근거**: 계획서 7절 전체 구현 순서
- **제안**:
  - MVP에서 서브관리자(Phase 1.5)를 제외하고, 메인관리자가 모든 학년을 직접 관리하는 구조로 시작. 서브관리자는 운영 후 필요성이 확인되면 추가.
  - 또는 학생 화면(Phase 4)에서 월간 달력 뷰를 제외하고 주간 출결 + 불참 신청만 MVP에 포함.
  - Phase 5의 테이블 뷰와 Excel 다운로드는 관리자에게 필수이므로 축소 비권장.

### 5-3. SWR 30초 폴링의 범위
- **심각도**: Suggestion
- **문제**: SWR 30초 폴링이 "모든 화면"에 필요한 것은 아니다. 감독교사 출석체크 화면에서만 실시간성이 필요하고, 관리자/담임/학생 화면은 온디맨드 새로고침으로 충분하다.
- **근거**: 계획서 6절 기술 스택 "SWR 30초 폴링"
- **제안**: 폴링 적용 범위를 명시하라:
  - 감독교사 출석체크: 30초 폴링 (다른 감독이 같은 학년을 볼 때 동기화)
  - 담임 불참신청 목록: 60초 또는 수동 새로고침
  - 학생 화면: 폴링 불필요 (자기 데이터만 조회)
  - 관리자 화면: 폴링 불필요 (데이터 양이 많아 비효율)

## 6. 기술적 위험

### 6-1. NextAuth.js에서 교사/학생 이중 Credentials Provider
- **심각도**: Critical
- **문제**: NextAuth.js v5에서 하나의 앱에 두 종류의 Credentials Provider (교사: ID/PW, 학생: 이름/학번)를 설정해야 한다. NextAuth는 기본적으로 단일 사용자 모델(`User`)을 가정하며, 교사(`Teacher`)와 학생(`Student`)이라는 별도 모델을 사용하려면 커스텀 어댑터를 작성하거나, `authorize` 콜백에서 분기 처리가 필요하다. JWT 토큰에 `userType: "teacher" | "student"`를 포함하고, 이후 모든 세션 체크에서 이를 분기해야 한다.
- **근거**: 계획서 5절 인증 "교사: ID/PW + Google", "학생: 이름(ID) + 학번(PW)", 6절 기술 스택 "NextAuth.js v5"
- **제안**:
  - Credentials Provider를 2개 등록 (id: "teacher-credentials", id: "student-credentials")하고, 로그인 페이지에서 탭에 따라 다른 Provider를 호출.
  - `jwt` 콜백에서 `token.userType`, `token.userId`, `token.roles` 등을 주입.
  - `session` 콜백에서 이를 세션에 반영.
  - 이 구현 패턴을 계획서에 구체적으로 기술하라 (Phase 0-5번 항목 확장).
  - Google Provider와의 조합: 교사가 Google로 최초 로그인 시 `Teacher.googleId` 매핑 로직도 필요.

### 6-2. Prisma enum 변경 시 마이그레이션 호환성
- **심각도**: Suggestion
- **문제**: 새 프로젝트이므로 기존 마이그레이션과의 호환성 문제는 없다. 다만, v3에서 v4로의 마이그레이션이 필요한 경우를 대비하여 `sub_admin` enum 추가에 대한 마이그레이션 전략이 있으면 좋다.
- **근거**: 계획서 3절 `enum Role`에 `sub_admin` 포함
- **제안**: 새 프로젝트이므로 현재는 문제없다. 단, 향후 enum 변경 시 PostgreSQL의 `ALTER TYPE ... ADD VALUE`가 트랜잭션 내에서 작동하지 않는 제약을 인지하고, Prisma의 `prisma migrate dev`가 이를 자동 처리하는지 확인할 것.

### 6-3. 메인관리자/서브관리자 컴포넌트 재사용 전략 부재 (기술적 관점)
- **심각도**: Warning
- **문제**: 3-5에서 언급한 것의 기술적 측면. `/admin/*`와 `/grade-admin/[grade]/*`가 Next.js App Router에서 별도 라우트 그룹이 되면, 레이아웃과 컴포넌트를 공유하기 어려울 수 있다. 특히 서버 컴포넌트에서 데이터 fetching 로직이 중복된다.
- **근거**: 계획서 4절 라우트 구조, 7절 Phase 1 vs Phase 1.5
- **제안**:
  - `app/(admin-shared)/` 라우트 그룹으로 공통 레이아웃 공유.
  - 또는 `app/admin/[...path]` catch-all 라우트 내에서 메인관리자/서브관리자를 분기하는 구조.
  - 데이터 fetching 함수는 `lib/queries/` 디렉토리에서 `grade` 파라미터를 받는 형태로 통합.

## 종합 평가

### 전반적 품질
계획서는 전체적으로 상세하고 체계적이다. 데이터 모델, API 설계, UI 목업, 구현 순서까지 잘 구조화되어 있다. v4에서 추가된 서브관리자, 학생 로그인, 불참 신청 기능의 요구사항은 명확히 정의되어 있다.

### Critical 이슈 (3건)
1. **AbsenceRequest 승인 → Attendance/AbsenceReason 연동 로직 미명시** (1-3): 핵심 비즈니스 로직이 빠져 있어 구현 시 혼란 초래
2. **서브관리자 학년 파라미터 조작 취약점** (2-1): URL 조작으로 타학년 데이터 접근 가능 -- 반드시 미들웨어에서 검증 필요
3. **NextAuth.js 이중 Credentials Provider 구현** (6-1): 기술적으로 가능하나 구현 패턴이 미정의 상태로, Phase 0에서 막힐 위험

### Warning 이슈 (8건)
sub_admin 이중 구조, 동명이인 로직 모호, 학생 로그인 보안, 감독교체 동의 부재, Excel 학번 자동생성, 코드 재사용 전략, 네비게이션 과부하, 미들웨어 복잡도

### Suggestion 이슈 (6건)
서브관리자 겸임 정책, 월간 달력 오후/야간 구분, MVP 범위, 폴링 범위, enum 마이그레이션, 학생 로그인 UX

### 실행 가능 여부
**조건부 실행 가능**. Critical 3건을 해결하면 구현을 시작할 수 있다. 특히 (1-3)과 (6-1)은 Phase 0 시작 전에 설계를 확정해야 하며, (2-1)은 Phase 0의 RBAC 미들웨어 구현 시 반영하면 된다. Warning 이슈들은 해당 Phase 진입 시 점진적으로 해결 가능하다.

### 추가 권장사항
- Phase 1과 Phase 1.5를 통합하여 학년별 컴포넌트 재사용 구조로 설계하면 전체 구현 시간을 20~30% 절감할 수 있다.
- 학생 로그인은 학번을 ID로 사용하는 방안을 재고하라 (학번이 유일한 식별자이므로 이름 입력의 오류 가능성 제거).
